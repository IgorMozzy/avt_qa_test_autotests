# Найденные дефекты

> Дефекты найдены в процессе исследования API и написания автотестов. Некоторые баги являются скорее открытыми вопросами, требующими согласования с проектными требованиями. В условиях тестирования API-сервиса баг-репорты сокращены до минимальных функциональных шагов, предполагающих знакомство читателей с механизмом отправки запросов и контекстом проекта. Каждый баг или "вопрос" сопровождается маркером `xfail` в pytest для актуализации после исправления либо исключения/корректировки автотестов после уточнения требований к проверкам.

**Окружение:** хост `https://qa-internship.avito.com`. Для HTTP-запросов используются заголовки: `Content-Type: application/json`, `Accept: application/json`.

## BUG-1: POST /item принимает отрицательную или переполняющую цену и возвращает 200 вместо 400

**Серьёзность:** Критическая

`POST /api/1/item` сохраняет объявление с отрицательной `price` или с очень большой ценой (`2**53+1`, в JSON удобно `9007199254740993`), ответ 200. Цена не может быть отрицательной; нужно зафиксировать в контракте верхнюю границу. При этом `price: 0` сервер отклоняет корректно (400).

**Шаги воспроизведения:**

1. Отправить `POST https://qa-internship.avito.com/api/1/item` с заголовками JSON.
2. Тело запроса (невалидная цена, остальное валидно):

```json
{
  "sellerID": 111111,
  "name": "Тест",
  "price": -100,
  "statistics": { "likes": 1, "viewCount": 1, "contacts": 1 }
}
```

3. Зафиксировать HTTP-код и при 200 - тело ответа.
4. Повторить шаги 1-2 с тем же телом, но `"price": 9007199254740993` (`2**53+1`), снова зафиксировать код ответа.

**Ожидаемый результат:** 400.

**Фактический результат:** 200, объявление создано.

**Затронутые тест-кейсы:** CREATE-14, CREATE-22

## BUG-2: POST /item принимает отрицательные значения статистики и возвращает 200 вместо 400

**Серьёзность:** Критическая

`POST /api/1/item` создаёт объявление, если в `statistics` отрицательный `likes`, `viewCount` или `contacts` (ответ 200). Счётчики не могут быть отрицательными, такой запрос должен получать 400.

**Шаги воспроизведения:**

1. Отправить `POST https://qa-internship.avito.com/api/1/item` с заголовками JSON.
2. Тело с отрицательным `likes` (аналогично проверяются `viewCount` и `contacts`):

```json
{
  "sellerID": 111111,
  "name": "Тест",
  "price": 1000,
  "statistics": { "likes": -1, "viewCount": 0, "contacts": 0 }
}
```

3. Зафиксировать HTTP-код (для `viewCount: -1` или `contacts: -1` - то же тело, меняется одно поле в `statistics`).

**Ожидаемый результат:** 400.

**Фактический результат:** 200.

**Затронутые тест-кейсы:** CREATE-19, CREATE-20, CREATE-21

## BUG-3: GET /:sellerID/item с отрицательным sellerID возвращает 200 вместо 400

**Серьёзность:** Средняя

`GET /api/1/{sellerID}/item` при отрицательном значении sellerID, например, `-111111` возвращает 200 и пустой массив `[]` либо список объявлений. Предполагается отклонение некорректного идентификатора продавца.

**Шаги воспроизведения:**

1. Отправить `GET https://qa-internship.avito.com/api/1/-111111/item` с заголовком `Accept: application/json`.
2. Зафиксировать HTTP-код и тело ответа.

**Ожидаемый результат:** 400.

**Фактический результат:** 200, `[]` (либо список объектов).

**Затронутые тест-кейсы:** SELLER-4

## BUG-4: DELETE /item/:id для несуществующего/удалённого ID - в теле `status: "500"` вместо `"404"`

**Серьёзность:** Средняя

`DELETE /api/2/item/:id` при повторном удалении или по несуществующему uuid отдаёт HTTP 404, но в JSON поле `status` содержит `"500"` вместо `"404"`, а `message` пустой. Ожидается наполнение тела ответа, согласованное с его кодом.

**Шаги воспроизведения:**

1. Создать объявление: `POST https://qa-internship.avito.com/api/1/item` с валидным JSON-телом, сохранить `id` объявления.
2. Удалить: `DELETE https://qa-internship.avito.com/api/2/item/{id}`.
3. Повторить `DELETE https://qa-internship.avito.com/api/2/item/{id}`.
4. Изучить JSON второго ответа: код, поля `result`, `status`, `message`.

**Ожидаемый результат:** HTTP 404, в теле `status: "404"`, осмысленный `message`.

**Фактический результат:** HTTP 404, `status: "500"`, `message: ""`.

**Затронутые тест-кейсы:** DEL-3, DEL-5

## BUG-6: POST /item возвращает строку статуса вместо объекта объявления

**Серьёзность:** Высокая

`POST /api/1/item` при успехе возвращает JSON вида `{"status":"Сохранили объявление - <uuid>"}`. Ответ не соответствует согласованной схеме, создает существенные сложности в использовании сервиса, затрудняет тестирование и отладку.

**Шаги воспроизведения:**

1. Отправить `POST https://qa-internship.avito.com/api/1/item` с валидным телом, например:

```json
{
  "sellerID": 111111,
  "name": "Тест",
  "price": 1000,
  "statistics": { "likes": 1, "viewCount": 1, "contacts": 1 }
}
```

2. Сравнить фактическую структуру JSON с ожидаемым объектом объявления.

**Ожидаемый результат:** Объект `{id, sellerId, name, price, statistics: {likes, viewCount, contacts}, createdAt}`.

**Фактический результат:** Объект вида `"status": "Сохранили объявление - <uuid>"`.

**Затронутые тест-кейсы:** CREATE-1 - CREATE-5, CREATE-5a

## BUG-7: GET /item/:id возвращает массив вместо объекта

**Серьёзность:** Низкая

`GET /api/1/item/:id` возвращает корень JSON как массив из одного объекта объявления, а не как один объект. Ответ соответствует предоставленной схеме, но не является типичным - требуется подтверждение корректности контракта.

**Шаги воспроизведения:**

1. Создать объявление через `POST https://qa-internship.avito.com/api/1/item`, получить `id`.
2. Выполнить `GET https://qa-internship.avito.com/api/1/item/{id}`
3. Проверить тип корневого значения в JSON: массив [] или объект {}.

**Ожидаемый результат:** Объект.

**Фактический результат:** Массив из одного объекта.

**Затронутые тест-кейсы:** GET-1, GET-5, E2E-1, E2E-2

## BUG-9: Поле createdAt с дублем таймзоны

**Серьёзность:** Средняя

В ответе `GET /api/1/item/:id` строка `createdAt` содержит дублирование смещения часового пояса (дважды подряд `+0300`), из-за чего значение нельзя разобратьобычным парсером дат без преобразований. Ожидается, что поле даты можно считывать и использовать стандартными инструментами.

**Шаги воспроизведения:**

1. Создать объявление через `POST https://qa-internship.avito.com/api/1/item`, получить `id`.
2. Выполнить `GET https://qa-internship.avito.com/api/1/item/{id}`.
3. В первом элементе массива проверить поле `createdAt` на дублирование смещения и возможность парсинга любым парсером дат.

**Ожидаемый результат:** Одна зона, формат ISO.

**Фактический результат:** Дублирование смещения таймзоны в строке.

**Затронутые тест-кейсы:** GET-1, GET-6, CREATE-1 - CREATE-5

## BUG-21: POST с неверным Content-Type - тело ошибки не по стандартной схеме

**Серьёзность:** Низкая

`POST /api/1/item` с телом JSON, но заголовком `Content-Type: text/xml` или `text/plain`, при ответе 400 отдаёт JSON с полями `message` и `code`, а не структуру `result` / `status` как у остальных ошибок сервиса.

**Шаги воспроизведения:**

1. Отправить `POST https://qa-internship.avito.com/api/1/item` с телом:
```json
{
  "sellerID": 111111,
  "name": "Пример",
  "price": 1000,
  "statistics": { "likes": 1, "viewCount": 1, "contacts": 1 }
}
```
и заголовками `Content-Type: text/xml` или `text/plain`
2. Разобрать JSON ответа и сравнить набор полей с типичной ошибкой API `{result: {messages: {}, message: ""}, status: ""}`

**Ожидаемый результат:** Схема как у остальных ошибок API.

**Фактический результат:** Ответ `{"message": "invalid content type", "code": 400}`

**Затронутые тест-кейсы:** HTTP-3, класс `TestUnsupportedContentType`

## BUG-23: POST с неверным типом поля - неинформативное тело ошибки

**Серьёзность:** Средняя

`POST /api/1/item` при неверном типе поля возвращает 400, но в JSON ответа присутствует пустой `message` и нестандартное значение `status`, нет явной ссылки на проблемное поле. Ожидается понятная и предсказуемая для клиента ошибка.

**Шаги воспроизведения:**

1. Отправить `POST https://qa-internship.avito.com/api/1/item` с заголовками JSON и телом, где тип поля неверный, например:

```json
{
  "sellerID": "not_a_number",
  "name": "Тест",
  "price": 1000,
  "statistics": { "likes": 1, "viewCount": 1, "contacts": 1 }
}
```

2. В JSON проверить поля `status` и `message`.

**Ожидаемый результат:** `status: "400"`, осмысленное `message`, отсылающее к проблемному полю.

**Фактический результат:** Пустой `message`, нестандартный `status` в теле. Пример:
```json
{
    "result": {
        "message": "",
        "messages": {}
    },
    "status": "не передано тело объявления"
}
```

**Затронутые тест-кейсы:** CREATE-26, CREATE-43
